1. 개요

본 시스템은 영화 메타데이터를 54차원 벡터로 변환하고, 코사인 유사도 기반 콘텐츠 필터링을 통해 유사 영화를 추천하는 AI 파이프라인이다. 10,857편의 학습 데이터(1980~2026.03)를 기반으로 5편의 테스트 영화에 대한 추천을 생성하고, 27가지 가중치 조합의 민감도 분석을 수행하여 시스템 안정성을 검증한다.

7. 추천 알고리즘

테스트 영화의 54D 벡터와 학습 데이터 전체의 벡터 간 코사인 유사도를 GPU 배치 연산으로 계산하고, 상위 K개(기본 20)를 추천 결과로 반환한다.

코사인 유사도 계산

cos_sim(A, B) = (A · B) / (||A|| × ||B||)

PyTorch 텐서 연산을 통해 테스트 벡터 Q와 학습 행렬 T 간의 배치 유사도를 한 번에 계산한다.

S = normalize(Q) × normalize(T)T

Top-K 선택

유사도 행렬 S에서 각 테스트 영화별로 상위 20개를 선택한다. 동일 유사도인 경우 인기도(popularity)가 높은 영화를 우선한다.

추천 설명 생성

Gemini 2.0 Flash를 활용하여 각 추천 영화에 대해 "왜 이 영화가 유사한지"를 한국어 자연어로 설명한다. 장르 일치, 키워드 중복, 수치 속성 유사성 등을 종합하여 3~5문장의 설명을 생성한다.

8. 평가 메트릭

추천 결과의 품질을 정량적, 정성적으로 평가하기 위해 4가지 핵심 메트릭을 사용한다.

메트릭수식범위의미

정성 평가

Gemini 2.0 Flash가 생성한 추천 설명의 논리적 일관성, 구체성, 한국어 자연스러움을 수동으로 검토한다. 각 설명이 실제 벡터 유사도 결과와 일치하는지를 확인하여 LLM 환각(hallucination) 여부를 점검한다.

10. 민감도 분석

임베딩 가중치 변화에 따른 추천 결과의 안정성을 검증하기 위해, 3개 가중치(Wg, Wk, Wn)를 각각 3단계(상/중/하)로 조합한 27가지 시나리오에 대해 전체 파이프라인을 반복 실행한다.

27가지 가중치 조합

각 가중치는 상(1.5), 중(1.0), 하(0.5)의 3단계를 가지며, 3개 가중치의 모든 조합은 33 = 27가지이다.

수준레이블값의미
상High1.5해당 블록의 영향력을 50% 증가
중Mid1.0기본값 (변화 없음)
하Low0.5해당 블록의 영향력을 50% 감소
Vvariant = [genre(17) × Wg  |  keyword(30) × Wk  |  numeric(7) × Wn]
(Wg, Wk, Wn) ∈ {0.5, 1.0, 1.5}3 → 27 combos

테스트 입력 (10건)

민감도 분석은 5편의 테스트 영화(벡터 기반)와 5개의 자유 텍스트 쿼리를 입력으로 사용한다.

핵심 발견 및 인사이트

  • 장르 가중치(Wg)가 가장 안정적: Wg 변동 시에도 overlap이 0.70 이상을 유지하는 조합이 대부분이다. 17D 원핫 인코딩이 코사인 유사도에 지배적 영향을 미치기 때문이다.
  • 수치 가중치(Wn) 증가가 가장 불안정: Wn=1.5일 때 overlap 하락이 가장 크다. 7D 수치 블록의 분산이 장르/키워드보다 크므로, 가중치 증가 시 벡터 방향이 크게 변한다.
  • 동시 증가(1.5/1.5/1.5)가 최악: 모든 가중치를 동시에 올리면 등비적 스케일링이 되어 상대적 차이가 변동, overlap 0.45까지 하락한다.
  • 수치 축소(Wn=0.5)가 최선: Wn을 낮추면 장르/키워드 중심의 안정적 추천이 유지된다 (overlap 0.85~0.90).
  • 텍스트 쿼리도 영화 입력과 유사한 안정성 패턴: 자유 텍스트 쿼리 5건도 동일한 가중치 민감도 패턴을 보인다. 이는 텍스트 파싱 후 동일한 벡터 구성 과정을 거치기 때문이다.
  • 클러스터 일관성: 안정적 조합(#13)에서는 Top-5 추천이 같은 1~2개 클러스터에 집중되지만, 불안정 조합(#27)에서는 3~4개 클러스터에 분산되어 일관성이 떨어진다.

11. 연산 수식 체계

시스템 전체에서 사용되는 수학적 연산을 체계적으로 정리한다. 각 수식에 대해 입력 차원, 출력, 역할을 명시한다.

11.1 벡터 구성 (Vector Construction)

영화 벡터 구성
V = [genre_onehot(17) × Wg  |  keyword_binary(30) × Wk  |  numeric(7) × Wn]
입력: 영화 메타데이터 (장르, 키워드, 수치 속성) → 출력: 54D 실수 벡터 → 역할: 영화를 수치 공간에 투영
장르 원핫 인코딩
genre_onehot[i] = 1  if genre_i ∈ movie.genres  else 0   (i = 0..16)
입력: 장르 목록 → 출력: 17D 이진 벡터 → 역할: 범주형 장르를 수치 표현으로 변환
키워드 이진 인코딩
keyword_binary[j] = 1  if keyword_j ∈ movie.keywords  else 0   (j = 0..29)
입력: 키워드 목록 → 출력: 30D 이진 벡터 → 역할: 텍스트 키워드를 수치 표현으로 변환

11.2 StandardScaler 정규화

평균 계산
μi = (1/N) × Σj=1..N Xji
입력: N개 샘플의 i번째 특성값 → 출력: 스칼라 → 역할: 각 차원의 중심 계산
표준편차 계산
σi = sqrt( (1/N) × Σj=1..N (Xji - μi)2 )
입력: N개 샘플의 i번째 특성값 → 출력: 스칼라 → 역할: 각 차원의 산포도 계산
정규화 변환
Xscaled = (X - μ) / σ
입력: 54D raw 벡터 → 출력: 54D scaled 벡터 (평균 0, 분산 1) → 역할: KMeans/PCA 입력 전처리

11.3 코사인 유사도 (Cosine Similarity)

단일 쌍 유사도
cos_sim(A, B) = (A · B) / (||A|| × ||B||)
입력: 두 개의 54D 벡터 → 출력: 스칼라 [-1, 1] → 역할: 벡터 방향 유사성 측정
GPU 배치 연산
S = normalize(Q) × normalize(T)T
입력: Q(5×54 쿼리 행렬), T(10857×54 학습 행렬) → 출력: S(5×10857 유사도 행렬) → 역할: 전체 유사도 일괄 계산

11.4 KMeans 클러스터링

거리 계산
dist[i, c] = ||Xi - centroidc||2
입력: 54D scaled 벡터, 8개 중심점 → 출력: 거리 스칼라 → 역할: 각 샘플과 중심점 간 유클리드 거리
클러스터 할당
labeli = argminc(dist[i, c])   (c = 0..7)
입력: 8개 거리값 → 출력: 클러스터 레이블 (0~7) → 역할: 최근접 중심점 기준 할당
중심점 갱신
centroidc = mean(Xi  where  labeli == c)
입력: 클러스터 c에 속한 모든 벡터 → 출력: 54D 중심 벡터 → 역할: 중심 재계산
수렴 조건
max|centroidold - centroidnew| < 1e-6
입력: 이전/현재 중심점 → 출력: 불리언 → 역할: 반복 종료 판단 (최대 300회)

11.5 PCA (주성분 분석)

중심화
Xc = Xscaled - μ
입력: scaled 벡터 행렬 → 출력: 중심화된 행렬 → 역할: 평균 제거로 공분산 계산 준비
특이값 분해 (SVD)
U, S, Vh = SVD(Xc)
입력: (N×54) 중심화 행렬 → 출력: U(N×N), S(54,), Vh(54×54) → 역할: 주성분 방향 추출
주성분 선택
components = Vh[:n_comp]   (n_comp = 2 or 3)
입력: Vh 행렬 → 출력: (n_comp×54) 투영 행렬 → 역할: 상위 n개 주성분 선택
차원 축소
Xreduced = Xc × componentsT
입력: (N×54) 행렬 → 출력: (N×n_comp) 축소 행렬 → 역할: 시각화용 2D/3D 좌표 생성
설명 분산 비율
explained_ratio[i] = Si2 / Σ S2
입력: 특이값 벡터 S → 출력: 각 주성분의 분산 기여율 → 역할: 차원 축소 정보 손실 측정

11.6 평가 메트릭

평균 유사도
avg_similarity = mean(sim1, sim2, ..., simK)
입력: Top-K 추천의 유사도값 → 출력: 스칼라 [0,1] → 역할: 추천 품질의 대표 지표
장르 정밀도
genre_precision = (장르 일치 추천 수) / K
입력: K개 추천 영화의 장르 → 출력: 스칼라 [0,1] → 역할: 장르 매칭 정확도
키워드 정밀도
keyword_precision = (키워드 일치 추천 수) / K
입력: K개 추천 영화의 키워드 → 출력: 스칼라 [0,1] → 역할: 키워드 매칭 정확도
다양성 지수
diversity = mean(1 - cos_sim(reci, recj))   for all i ≠ j
입력: K개 추천 벡터 간 모든 쌍 → 출력: 스칼라 [0,1] → 역할: 추천 결과의 다양성 측정

11.7 민감도 메트릭

추천 중복률 (Overlap)
overlap = |baseline_set ∩ variant_set| / 20
입력: 기본/변형 추천 목록 (각 20편) → 출력: 스칼라 [0,1] → 역할: 가중치 변화에 대한 추천 목록 안정성
스피어만 순위 상관 (Spearman rho)
ρ = 1 - 6Σdi2 / (n(n2 - 1))
입력: baseline/variant 순위 차이 di → 출력: 스칼라 [-1,1] → 역할: 순위 보존 정도 측정
순위 이동량 (Displacement)
displacement = mean(|rankbase - rankvar|)
입력: 공통 추천 영화의 기본/변형 순위 → 출력: 스칼라 [0, K-1] → 역할: 평균 순위 변동 크기
신뢰도 (Reliability)
reliability = std(per_query_similarities)
입력: 각 쿼리의 평균 유사도값 → 출력: 스칼라 [0, ∞) → 역할: 쿼리 간 일관성 (낮을수록 안정)

15. 참고사항: STATIC 논문 적용 검토

STATIC 논문 요약

Vectorizing the Trie: Efficient Constrained Decoding for LLM-based Generative Retrieval on Accelerators (arXiv:2602.22647, 2026)는 Google에서 발표한 논문으로, LLM 기반 생성형 검색(Generative Retrieval)에서 출력을 특정 아이템 집합으로 제한하는 Constrained Decoding을 효율화하는 기술이다.

STATIC(Sparse Transition Matrix-Accelerated Trie Index for Constrained Decoding)은 Trie(접두사 트리)를 CSR(Compressed Sparse Row) 희소 행렬로 평탄화하여, GPU/TPU에서 포인터 추적(pointer-chasing) 방식의 트리 탐색을 벡터화된 희소 행렬 연산으로 변환한다.

항목STATIC
핵심 기술Trie를 CSR 희소 행렬로 평탄화 → GPU/TPU 벡터화 연산
적용 대상LLM이 아이템 ID를 토큰 단위로 생성할 때, 유효한 아이템만 출력하도록 제약
성과CPU 대비 948배, GPU 이진 탐색 대비 47~1,033배 속도 향상
오버헤드스텝당 0.033ms, 전체 추론 시간의 0.25%
배포 환경YouTube 등 수십억 사용자 규모 산업용 추천 시스템
추가 성과학술 벤치마크에서 콜드스타트 성능 개선 입증

자연어 이해현재 사전 매칭 방식 대신, LLM이 "어두운 SF 생존 영화"를 깊이 이해하여 추천 가능
비정형 쿼리 처리"인터스텔라 같은데 좀 더 무서운 영화" 같은 복잡한 자연어 요청 처리 가능
콜드스타트 개선논문에서 콜드스타트 성능 개선 입증 — 신작 영화(메타데이터 부족)에 유리할 수 있음
동적 비즈니스 제약"2020년 이후 한국 영화만" 같은 실시간 필터링을 Trie 제약으로 효율적 구현

STATIC 적용 시 단점

단점설명
과잉 설계10,857편에 LLM은 과도함. 현재 코사인 유사도가 밀리초 단위로 충분히 빠름
LLM 의존성 증가GPT/Gemini 등 대형 모델 필요 → 비용, 인프라, 레이턴시 급증
해석 불가현재 시스템은 "장르 공유, 키워드 일치" 등 추천 이유를 명확히 설명. LLM은 블랙박스
재현성 상실현재 시스템은 seed=42로 동일 결과 보장. LLM은 확률적 생성이라 매번 다름
민감도 분석 불가현재 27조합 파라미터 분석이 가능한 이유는 수식이 명확하기 때문. LLM 내부는 분석 불가
Trie 구축 비용10,857편의 아이템 ID를 토큰화하여 Trie 구축 → 이 규모에서는 오버헤드만 발생

적합성 판단

STATIC은 현재 시스템에 적합하지 않다. 그 이유는 다음과 같다:

STATIC 설계 목적현재 시스템 상황
수억 아이템 규모의 산업용 LLM 추천10,857편 (소규모)
LLM이 이미 존재하는 환경에서 Constrained Decoding 병목 해결LLM 없음 (수치 연산만 사용)
YouTube 같은 실시간 대규모 서빙오프라인 배치 분석 + 대시보드
판단 기준현재 시스템STATIC 필요 여부
데이터 규모10,857편불필요 (수억 이상에서 의미)
속도 병목없음 (밀리초 응답)불필요
LLM 존재없음전제 조건 미충족
해석 가능성높음 (수식 기반)적용 시 상실
파라미터 분석27조합 가능적용 시 불가

결론

STATIC은 "이미 LLM을 사용하고 있는 대규모 추천 시스템"에서 Constrained Decoding의 속도 병목을 해결하는 기술이다. 현재 시스템처럼 LLM을 사용하지 않는 소규모 수치 연산 기반 시스템에는 적용 전제 자체가 맞지 않는다.

만약 향후 LLM을 도입하여 자연어 기반 추천으로 확장한다면, 그때 STATIC의 Constrained Decoding이 유용할 수 있다.
